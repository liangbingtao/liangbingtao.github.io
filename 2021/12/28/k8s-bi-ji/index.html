K8s

概念

  1. 什么是Ｋ８ｓ

    Ｋ８ｓ是一组服务器集群，Ｋ８ｓ所管理的是集群节点上的容器

  2. Ｋ８ｓ的功能

    • 自我修复
    • 弹性伸缩：事实根据服务器的并发情况，增加或缩减容器数量
    • 自动部署
    • 回滚
    • 服务发现和负载均衡
    • 机密和配置共享管理

  3. Ｋ８ｓ集群分为两类节点

    • ｍａｓｔｅｒ　ｎｏｄｅ：主节点
    • ｗｏｒｋｅｒ　ｎｏｄｅ：工作节点

  4. ｍａｓｔｅｒ　节点的组件（程序）

    • ａｐｉｓｅｒｖｅｒ：接收客户端操作K8s的指令，Kubernetes API,集群的统一入口，各组件协调者，以RESTful API提供接口服务，所有对象资源的增删改查和监听操作都交给APIServer处理后在提交给Etcd存储。
    • ｓｃｈｄｕｌｅｒ：调度器，从多个worker node节点的组件中选举一个来启动服务。根据调度算法为新创建的Pod选择一个Node节点，可以任意部署，可以部署在同一个节点上也可以部署在不同的节点上。
    • ｃｏｎｔｒｏｌｌｅｒ　ｍａｎａｇｅｒ：向worker节点的kubelet发送指令的。处理集群中常规后台任务，一个资源对应一个控制器，而ControllerManager就是负责管理这些控制器的。
    • etcd数据库：K8s的一个分布式键值数据库，用来注册节点、服务、记录账户等操作。分布式键值存储系统，用于保存集群状态数据，如Pod、Sercice等对象信息。

  5. ｎｏｄｅ节点的程序

    • kubelet:向docker发送指令管理docker容器的。Kubelet是Master在Node节点上的Agent,管理本机运行容器的生命周期，比如创建容器、Pod挂载数据卷，下载secret、获取容器和节点状态等工作。kubelet将每个pod转换成一组容器。
    • kubeproxy：管理docker容器的网络。在node节点上实现pod网络代理，维护网络规则和四层负载均衡工作。
    • docker或rocket：容器引擎，运行容器。

  6. ==Pod==

    • K8s中最小的部署单元，一个pod中可以有一个或者多个容器，pod又称为容器组（一组容器的集合）
    • 一个pod中的容器共享网络命名空间。pod是短暂的。
    • K8s中不可以直接启动容器

  7. Controllers:控制器控制pod,启动，删除，停止

    • ReplicaSet:确保预期pod副本的数量
    • ==Deployment:==无状态应用部署
    • Statefulset:有状态应用部署
    • DaemonSet:确保所有node运行同一个pod
    • Job：一次性任务
    • Cronjob:定时任务

    pod 与pod控制器:

    一个pod中是一个或者多个容器，pod控制器是用来控制，管理pod的数量，状态

  8. Service:服务

    • 将一组pod关联起来，提供一个统一的入口，即使pod地址发生改变，这个统一的入口也不会变化，可以保证用户访问不受影响(防止pod失联，定义一组pod的访问策略)

  9. label：标签

    • 附加到某个资源上，用于关联对象查询和筛选。一组pod有一个统一的标签，service是通过标签和一组pod进行关联的

  10. namespace：命名空间，用来隔离pod的运行环境，【默认情况下pod是可以互 相访问的】

    使用场景：

    • 为不同的公司提供隔离的pod运行环境
    • 为开发环境、测试环境、生产环境分别准备不同的命名空间进行业务隔离

==部署etcd==

etcd需要三台虚拟机独立部署，这里单master集群我们在master，node1,node2上分别安装一个etcd

首先在root下新建k8sFiles文件夹，在文件夹中上传etcd.tar.gz安装包，并且将其拷贝到“/root/”目录下,并解压

​ #cp etcd.tar.gz /root/

​ #tar xvf etcd.tar.gz

解压之后会生成一个文件etcd.service（systemd服务管理脚本）和一个目录etcd

systemd 服务管理脚本放在“/usr/lib/systemd/system”下

​ #mv etcd.service /usr/lib/systemd/system/

将etcd目录移动到”/opt/”目录下

​ #mv etcd /opt/

修改etcd的配置文件

​ #vim /opt/etcd/cfg/etcd.conf

修改之后：

将证书文件拷贝到etcd的目录下：

#cp /root/TLS/etcd/{ca,server,server-key}.pem /opt/etcd/ssl

到此，master节点的etcd配置完成。

由于master配置已经完成，因此我们可以直接拷贝配置好的文件到另外两个节点：将服务管理脚本拷贝到node1对应目录下，拷贝到node2同理。

==第一步：安装配置docker==

把master节点上k8sFiles文件夹中的k8s-node.tar.gz压缩包给node1和node2分别发一份
scp k8s-node.tar.gz root@k8s-node1:/root/
scp k8s-node.tar.gz root@k8s-node2:/root/


在node1节点上对该压缩包解压
tar xvf k8s-node.tar.gz

解压产生了三个文件，daemon.json是docker的配置文件，docker.service是docker的服务脚本，docker-18.09.6.tgz是docker的二进制程序。

mv docker.service /usr/lib/systemd/system   #把服务管理脚本移动到/usr/lib/systemd/system中
mkdir /etc/docker		#/etc目录下创建docker配置文件目录
cp daemon.json /etc/docker	#将docker的配置文件复制到该文件夹中
tar xf docker-18.09.6.tgz	#解压缩docker的二进制程序，会产生一个docker文件夹，文件夹中是docker所有的管理程序，我们把它放到path路径可以找到的位置，我们放在了/bin/目录下
mv docker/* /bin/

systemctl start docker  #启动docker
systemctl enable docker #开机自启动
docker info #查看docker 信息，需要注意的是，每个版本的k8s对所支持的docker版本也有要求，可以到github或官网查看

==第九：远程管理k8s==

默认情况下，k8s仅仅可以在master节点进行管理操作。在worker节点上，即使有工具也不能进行管理。

1） 将管理工具复制到worker节点：

master节点执行：
scp /bin/kubectl root@k8s-node1:/bin
scp /bin/kubectl root@k8s-node2:/bin
注：scp 命令用于在linux之间进行目录和文件的复制，而cp用于在同一个linux上进行文件的复制

2） 生成管理员证书：在master节点执行

# cd TLS/k8s
# vim admin-csr.json
写入如下内容：
        {
          "CN": "admin",
          "hosts": [],
          "key": {
            "algo": "rsa",
            "size": 2048
          },
          "names": [
            {
              "C": "CN",
              "L": "BeiJing",
              "ST": "BeiJing",
              "O": "system:masters",
              "OU": "System"
            }
          ]
        }

颁发admin证书：
# cfssl gencert -ca=ca.pem -ca-key=ca-key.pem -config=ca-config.json -profile=kubernetes admin-csr.json | cfssljson -bare admin

3） 创建kubeconfig文件

​ 设置集群参数

kubectl config set-cluster kubernetes \
  --server=https://192.168.177.63:6443 \
  --certificate-authority=ca.pem \
  --embed-certs=true \
  --kubeconfig=config

设置客户端认证参数

kubectl config set-credentials cluster-admin \
  --certificate-authority=ca.pem \
  --embed-certs=true \
  --client-key=admin-key.pem \
  --client-certificate=admin.pem \
  --kubeconfig=config

设置上下文参数

kubectl config set-context default \
  --cluster=kubernetes \
  --user=cluster-admin \
  --kubeconfig=config

设置默认上下文

kubectl config use-context default --kubeconfig=config

完成之后会在k8s目录下生成config文件，文件中就是生成的秘钥，我们需要将config发送到worker节点，当worker节点在实现kubectl时会读取config中的内容，将秘钥拿出来使用秘钥向master发送请求做认证，认证通过后就可以管理整个集群。

4） 将生成的config文件发送到worker节点[在master节点执行]

scp config root@k8s-node2:/root/

5） 在worker节点，基于config实现执行kubectl命令

kubectl get nodes --kubeconfig=/root/config

加上config文件可以看到和master节点执行时一样的效果，但是每次都要加上这个参数很麻烦，

mkdir ~/.kube  #root路径下创建.kube文件夹

方法3：通过yaml方式

我们在tmp目录下创建一个yaml文件ngx-dep.yaml，在yaml文件中编写如下内容：

    apiVersion: apps/v1			#指定api版本号
    kind: Deployment			#创建对象的类型，创建了一个deployment 
    metadata:					#deployment的信息，包括名称和标签
      name: ngx-dep3
      labels:
          app: ngx
          type: webservice
    spec:						#通过deployment创建出来的pod的信息
      replicas: 1				#副本数量
      selector:					#使用标签选择器
        matchLabels:
          app: ngx
      template:					#基于什么模板创建pod
        metadata:
          labels:
            app: ngx
        spec:					#创建出的容器的名称和使用的镜像 
          containers:
          - name: nginx
            image:nginx: 1.8

3)发布应用

​ 用service来发布服务

  • ​ 通过yaml方式发布
在master节点的/tmp目录下创建ngx-service.yaml 文件，并在文件中写入下面的内容
    apiVersion: v1
    kind: Service
    metadata:			#元数据
      name: ngx-svc		#service的名称
      labels:
        app: ngx
    spec:				#当前的service是管理哪一类pod，具有app标签并且值为ngx,通过选择器实现
      selector:
        app: ngx
      ports:			#通过ports将pod的端口暴露给外部
      - name: nginx-ports
        protocol: TCP		#采用的协议
        port: 80			
        nodePort: 32002		#将具有ngx标签的80端口暴i露到32002端口，这个端口要在30000~32767之间
        targetPort: 80		#将访问端口32002的请求转发到pod内部的80端口
      type: NodePort

执行yaml 文件：
kubectl apply -f ngx-service.yaml

执行完成之后可以在master上看到端口映射信息：

kubectl get svc
